# Contributing to Water Utility Ontology

Thank you for your interest in contributing to the Water Utility Ontology project. This document provides guidelines for participation.

## Ways to Contribute

### 1. Feedback and Discussion
- Open an [Issue](../../issues) for bugs, suggestions, or questions
- Join [Discussions](../../discussions) for broader topics
- Comment on existing issues and pull requests

### 2. Documentation
- Fix typos or clarify existing documentation
- Add examples showing real-world usage
- Translate standards for international use

### 3. Standards Development
- Propose new standards via the RFC process
- Review and comment on draft standards
- Pilot implementations and share results

### 4. Code and Tools
- Build reference implementations
- Create validation tools
- Develop migration utilities

## Standards Development Process

### Stages

1. **Proposal** - Initial idea submitted as an Issue with `[RFC]` prefix
2. **Draft** - Working group develops specification
3. **Review** - Community feedback period (minimum 30 days)
4. **Pilot** - Real-world testing at participating utilities
5. **Stable** - Approved for production use

### Working Groups

Each standard area has a working group:
- Asset Naming Working Group
- Data Hierarchy Working Group
- SCADA Visualization Working Group

To join a working group, comment on the relevant issue or reach out via Discussions.

## Pull Request Process

1. **Fork** the repository
2. **Create a branch** for your changes (`feature/your-feature-name`)
3. **Make changes** following the style guidelines below
4. **Test** your changes if applicable
5. **Submit PR** with clear description of changes

### Style Guidelines

**Markdown**
- Use ATX-style headers (`#`, `##`, etc.)
- One sentence per line for easier diffs
- Use fenced code blocks with language identifiers

**Standards Documents**
- Follow the template in `docs/standard-template.md`
- Include rationale for design decisions
- Provide examples for all specifications

## Code of Conduct

- Be respectful and inclusive
- Focus on the work, not the person
- Assume good intentions
- No vendor promotion or sales pitches

## Questions?

- Open a [Discussion](../../discussions) for general questions
- Tag maintainers in issues for urgent matters
- Email: standards@evolvewater.ai
